PPT에 넣었던 DB 1차 설계
--------------------------------------------------------------------------------------------
USER DB - 각 사용자가 실제로 사용할 정보
	- 계정정보, 개인화 정보
	- 세션 목록, 세션 주제, 대화내용
	- 업로드 및 참고 파일, 만들어진 여행 일정 자체
 	- 등...

API DB - 실시간 데이터 쿼리를 위한 API 풀
	- 기상청, 교통상황 등 여러 API 주소
	- 검색에 주로 사용되는 입력 데이터 형식
	- API 메타데이터
	- 등...

CSV DB - 정적 데이터 쿼리를 위한 테이블
	- 도로망 
	- 카페, 맛집, 펜션, 주요 관광지 
	- 등...


--------------------------------------------------------------------------------------------
DB에 어떤 테이블에 어떤 정보가 필요한지
--------------------------------------------------------------------------------------------
A. 사용자 관련
사용자 정보, 선호도, 저장한 여행

B. 관광 데이터 관련
관광지, 맛집, 숙소, 카페, 카테고리, 위치, 운영시간 등

C. 여행 일정 관련
여행 계획, 일자별 일정, 방문 순서

D. 외부 데이터 수집 관련
API/크롤링 원본, 업데이트 날짜, 출처

E. AI 처리 관련
LLM이 만든 추천 결과, 프롬프트 결과, 요약 정보

꼭 필요한 테이블들
--------------------------------------------------------------------------------------------
1) users
사용자 기본 정보

들어갈 정보
1. user_id
2. name 또는 nickname
3. email
4. created_at

왜 필요?
로그인 기능, 사용자별 여행 일정 저장, 즐겨찾기 저장용.

--------------------------------------------------------------------------------------------
2) user_preferences
사용자 여행 성향/선호 저장

들어갈 정보
1. preference_id
2. user_id
3. travel_style
	예: 감성, 맛집 중심, 자연 중심, 액티비티 중심
4. transport_type
	예: rental_car / bus / taxi / walk
5. preferred_food
6. preferred_schedule_density
	예: 빡빡한 일정 / 여유로운 일정
7. companion_type
	예: 혼자 / 친구 / 가족 / 커플

왜 필요?
AI가 추천할 때 사용자 맞춤형으로 일정 생성 가능.

3) places
가장 핵심 테이블.
관광지, 맛집, 카페, 숙소 등을 저장하는 메인 테이블.

들어갈 정보
1. place_id
2. name
3. place_type
	예: tourist_spot / restaurant / cafe / hotel / activity
4. description
5. address
6. region
	예: 제주시 / 서귀포시
7. latitude (위도)
8. longitude
9. phone (전화번호)
10. website_url (웹사이트)
11. average_rating (평균평점)
11. price_level (가격대)
12. indoor_outdoor (실내 실외 둘다)
13. recommended_stay_minutes (체류시간)
14. is_active 
15. created_at
16. updated_at

왜 이렇게 하냐?
관광지/맛집/숙소를 각각 따로 테이블로 쪼갤 수도 있는데,
이번 학기 프로젝트에서는 하나의 places 테이블로 통합하는 게 훨씬 현실적이야.

즉,
성산일출봉도 places
흑돼지 맛집도 places
호텔도 places
이렇게 관리하면 검색/추천/지도 표시가 편해져.

한 명의 user 는 여러 개의 trip 을 가질 수 있음
한 개의 trip 은 여러 개의 trip_day 를 가짐
한 개의 trip_day 는 여러 개의 itinerary_item 을 가짐
한 개의 itinerary_item 은 하나의 place 를 참조
한 개의 place 는 여러 개의 category 를 가질 수 있음
한 개의 place 는 여러 개의 hours, images 를 가질 수 있음
한 개의 place 는 하나 이상의 data_source 와 연결될 수 있음

데이터베이스 설계는 여행 관련 정보를 효과적으로 관리하고 사용자 맞춤형 경험을 제공하는 데 중요한 역할을 합니다. 아래는 제안된 데이터베이스 테이블의 요약입니다.

### 사용자 관련 데이터
1. **users**: 사용자 기본 정보 저장
   - user_id, name 또는 nickname, email, created_at

2. **user_preferences**: 사용자 여행 성향 및 선호 저장
   - preference_id, user_id, travel_style, transport_type, preferred_food, preferred_schedule_density, companion_type

### 관광 데이터 관련
3. **places**: 관광지, 맛집, 카페, 숙소 등 저장
   - place_id, name, place_type, description, address, region, latitude, longitude, phone, website_url, average_rating, price_level, indoor_outdoor, recommended_stay_minutes, is_active, created_at, updated_at

4. **place_categories**: 장소의 세부 카테고리 저장
   - category_id, category_name

5. **place_category_map**: 장소와 카테고리 간의 연결 테이블
   - place_id, category_id

6. **place_hours**: 운영시간 저장
   - hour_id, place_id, day_of_week, open_time, close_time, is_closed

7. **place_images**: 장소 이미지 저장
   - image_id, place_id, image_url, source, is_main

### 여행 일정 관련
8. **trips**: 사용자의 여행 계획 전체 단위 저장
   - trip_id, user_id, title, destination, start_date, end_date, num_people, companion_type, transport_type, status, created_at

9. **trip_days**: 여행의 날짜 단위 저장
   - trip_day_id, trip_id, day_number, date

10. **itinerary_items**: 실제 일정 항목 저장
    - item_id, trip_day_id, place_id, visit_order, start_time, end_time, memo, transport_from_prev, travel_time_from_prev

### 외부 데이터 수집 관련
11. **data_sources**: 데이터 출처 관리
    - source_id, source_name, source_type, api/crawling/manual, base_url, last_collected_at

12. **place_source_map**: 장소별 데이터 출처 연결
    - place_id, source_id, external_source_key, collected_at

### AI 처리 관련
13. **reviews_summary**: 리뷰 요약 저장
    - summary_id, place_id, summary_text, positive_keywords, negative_keywords, review_count, updated_at

### 날씨 및 계절성 정보
14. **weather_snapshot**: 날씨 정보 저장
    - weather_id, region, date, condition, temp_min, temp_max, rain_probability

15. **seasonal_tags**: 계절성 정보 저장
    - seasonal_tag_id, place_id, season, reason

이 데이터베이스 설계는 사용자의 여행 경험을 개인화하고, 다양한 관광 관련 데이터를 효율적으로 관리할 수 있도록 돕는 구조로 되어 있습니다. 각 테이블 간의 관계를 잘 설정하면 데이터의 일관성을 유지할 수 있습니다.

----



해당 테이블에 대해서, 각 테이블의 값들을 최적화 및 정합하게 정리하고, 각 테이블이 어떤 테이블과 조인 및 연결이 될지 관계를 산정하라.

[봇]
제안된 데이터베이스 테이블에 대한 최적화 및 정합성을 고려하여 각 테이블의 구조를 재정리하고, 테이블 간의 관계를 정의하겠습니다.

### 1. users
- **필드**: user_id, name/nickname, email, created_at, birthday
- **설명**: 사용자 기본 정보를 저장하며, 사용자 식별 및 로그인 기능에 사용됩니다.

### 2. user_preferences
- **필드**: preference_id, user_id, travel_style, transport_type, preferred_food, preferred_schedule_density, companion_type
- **설명**: 사용자의 여행 성향 및 선호 정보를 저장합니다.
- **관계**: user_id는 users 테이블의 user_id와 연결됩니다.

### 3. places
- **필드**: place_id, name, place_type, description, address, region, latitude, longitude, website_url, indoor_outdoor, created_at, updated_at
- **설명**: 관광지, 맛집, 카페, 숙소 등 다양한 장소 정보를 통합하여 저장합니다.

### 4. place_categories
- **필드**: category_id, category_name
- **설명**: 장소의 세부 카테고리를 저장합니다.

### 5. place_category_map
- **필드**: place_id, category_id
- **설명**: 장소와 카테고리 간의 다대다 관계를 설정합니다.
- **관계**: place_id는 places 테이블의 place_id와 연결되며, category_id는 place_categories 테이블의 category_id와 연결됩니다.

### 6. place_hours
- **필드**: hour_id, place_id, day_of_week, open_time, close_time, is_closed
- **설명**: 장소의 운영 시간을 요일별로 저장합니다.
- **관계**: place_id는 places 테이블의 place_id와 연결됩니다.

### 7. place_images
- **필드**: image_id, place_id, image_url, source, is_main
- **설명**: 장소에 대한 이미지를 저장합니다.
- **관계**: place_id는 places 테이블의 place_id와 연결됩니다.

### 8. trips
- **필드**: trip_id, user_id, title, destination, start_date, end_date, num_people, companion_type, transport_type, status, created_at
- **설명**: 사용자의 여행 계획 정보를 저장합니다.
- **관계**: user_id는 users 테이블의 user_id와 연결됩니다.

### 9. trip_days
- **필드**: trip_day_id, trip_id, day_number, date
- **설명**: 여행의 날짜별 정보를 저장합니다.
- **관계**: trip_id는 trips 테이블의 trip_id와 연결됩니다.

### 10. itinerary_items
- **필드**: item_id, trip_day_id, place_id, visit_order, start_time, end_time, memo, transport_from_prev, travel_time_from_prev
- **설명**: 각 일정의 구체적인 항목을 저장합니다.
- **관계**: trip_day_id는 trip_days 테이블의 trip_day_id와 연결되며, place_id는 places 테이블의 place_id와 연결됩니다.

### 11. data_sources
- **필드**: source_id, source_name, source_type, api/crawling/manual, base_url, last_collected_at
- **설명**: 데이터 출처 정보를 관리합니다.

### 12. place_source_map
- **필드**: place_id, source_id, external_source_key, collected_at
- **설명**: 장소별 데이터 출처 정보를 연결합니다.
- **관계**: place_id는 places 테이블의 place_id와 연결되며, source_id는 data_sources 테이블의 source_id와 연결됩니다.

### 13. reviews_summary
- **필드**: summary_id, place_id, summary_text, positive_keywords, negative_keywords, review_count, updated_at
- **설명**: 장소에 대한 리뷰 요약 정보를 저장합니다.
- **관계**: place_id는 places 테이블의 place_id와 연결됩니다.

### 14. weather_snapshot
- **필드**: weather_id, region, date, condition, temp_min, temp_max, rain_probability
- **설명**: 특정 지역의 날씨 정보를 저장합니다.

### 15. seasonal_tags
- **필드**: seasonal_tag_id, place_id, season, reason
- **설명**: 장소에 대한 계절성 정보를 저장합니다.
- **관계**: place_id는 places 테이블의 place_id와 연결됩니다.

### 테이블 간의 관계 요약
- users <-> user_preferences: 1:N
- users <-> trips: 1:N
- trips <-> trip_days: 1:N
- trip_days <-> itinerary_items: 1:N
- places <-> place_category_map: 1:N
- place_categories <-> place_category_map: 1:N
- places <-> place_hours: 1:N
- places <-> place_images: 1:N
- places <-> place_source_map: 1:N
- data_sources <-> place_source_map: 1:N
- places <-> reviews_summary: 1:N
- places <-> seasonal_tags: 1:N

이러한 구조와 관계를 통해 데이터의 일관성을 유지하고, 사용자 맞춤형 여행 계획을 효과적으로 지원할 수 있습니다.
