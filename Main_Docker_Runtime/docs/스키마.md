팀(Team) 도메인을 신설하여 개인 및 그룹 여행을 통합 관리할 수 있도록 구조화했습니다. 여행(Trip)과 세션(Session) 간의 1:N 관계를 확립하고, '제안-선택' 플래닝 메커니즘과 알림 시스템을 데이터베이스 레벨에서 지원합니다. 공간 데이터 연산을 위해 PostGIS의 지리 데이터 타입을 적용했습니다.

# 통합 데이터베이스 스키마 설계서 (팀/제안 플래닝 확장본)

사용자 개인의 정보와 다중 접속/권한 관리를 위한 팀 단위 테이블입니다. 개인 여행의 경우 1인 팀으로 취급하여 로직을 단일화할 수 있습니다.

### `users` — 마스터 식별자
| 칼럼명 | 타입 | NULL | 키 | 기본값 | 설명 |
|--------|------|------|----|--------|------|
| user_id | VARCHAR(40) | NO | PK | — | 사용자 고유 식별자. 접두사로 유형 내장: `MEM:{uuid}`, `GST:{uuid}`, `KKO:{hash16}` |
| user_type | VARCHAR(10) | NO | — | — | 접두사 파싱 없이 즉시 유형 조회용: `MEM`, `GST`, `KKO` |

| created_at | TIMESTAMPTZ | NO | — | NOW() | 계정 최초 생성 시각 |

| 칼럼명 | 타입 | NULL | 키 | 기본값 | 설명 |

### `user_profile` — 기본 프로필 및 연락 정보 (1:1)

## 2. 여행 기획 및 세부 일정 도메인 (Trips & Itineraries)
팀이 소유하며, 일자별/단위별로 세밀하게 쪼개진 정보(위치, 시간, 날씨, 메모)를 담습니다.

### `trips` — 여행 마스터 (1개의 여행)
| 칼럼명 | 타입 | NULL | 키 | 기본값 | 설명 |
|--------|------|------|----|--------|------|
| trip_id | VARCHAR(50) | NO | PK | — | 여행 고유 식별자 |
| team_id | VARCHAR(50) | NO | FK | — | 해당 여행을 관리하는 주체 (`teams` 참조) |
| created_by | VARCHAR(40) | NO | FK | — | 여행을 최초 생성한 사용자 (`users` 참조) |
| title | VARCHAR(255) | NO | — | — | 여행 프로젝트 이름 |
| destination | VARCHAR(100) | NO | — | — | 최종 목적지 (예: 제주도) |
| start_date | DATE | NO | — | — | 여행 시작일 |
| end_date | DATE | NO | — | — | 여행 종료일 |
| expected_budget | DECIMAL(15,2)| YES | — | — | 예상 총 예산 |
| cover_img_url | TEXT | YES | — | — | 여행 대표 커버 이미지 URL |
| status | VARCHAR(20) | NO | — | `planning` | `planning`, `confirmed`, `traveling`, `completed` |
| created_at | TIMESTAMPTZ | NO | — | NOW() | 생성 시각 |
| updated_at | TIMESTAMPTZ | NO | — | NOW() | 최종 갱신 시각 |

### `trip_days` — 여행 일자 (N일차)
`(trip_id, day_number)` 조합에 UNIQUE 제약을 적용하여 동일 여행에 중복 일차가 생성되는 것을 방지합니다.

| 칼럼명 | 타입 | NULL | 키 | 기본값 | 설명 |

| trip_id | VARCHAR(50) | NO | FK | — | `trips` 참조 |
| day_number | INT | NO | UNIQUE(trip_id, day_number) | — | 1일차, 2일차 등 순서. 여행 내 중복 불가 |
| target_date | DATE | NO | UNIQUE(trip_id, target_date) | — | 실제 캘린더 상의 날짜. 여행 내 중복 불가 |
| weather_id | VARCHAR(50) | YES | FK | — | 해당 일자 날씨 스냅샷 (`weather_snapshots` 참조) — 날씨는 일자 단위이므로 day 레벨에서 관리 |
| daily_memo | TEXT | YES | — | — | 해당 일자에 대한 전체 요약 메모 |

### `itinerary_items` — 단위별 세부 일정 (방문 장소, 날씨, 지도 맵핑)
| 칼럼명 | 타입 | NULL | 키 | 기본값 | 설명 |
|--------|------|------|----|--------|------|
| item_id | VARCHAR(50) | NO | PK | — | 일정 단위 고유 식별자 |
| day_id | VARCHAR(50) | NO | FK | — | `trip_days` 참조 |
| place_id | VARCHAR(50) | YES | FK | — | 실제 방문할 장소 (`places` 참조, 이동 전용 일정일 경우 NULL) |
| proposal_id | VARCHAR(50) | YES | FK | — | 이 일정을 생성한 제안 (`proposals` 참조) — 수락된 제안으로부터 생성된 경우에만 존재 |
| visit_order | INT | NO | — | — | 당일 내에서의 일정 순서 |
| start_time | TIME | YES | — | — | 시작/방문 예정 시간 |
| end_time | TIME | YES | — | — | 종료/이동 예정 시간 |
| transport_mode | VARCHAR(20) | YES | — | — | 이전 일정→현 일정 이동 수단: `walk`, `taxi`, `bus`, `rental_car`, `subway` |
| map_route_data| JSONB | YES | — | — | 이전 일정에서 현 일정으로의 이동 경로, 폴리라인, 거리 등의 지도 데이터 |
| estimated_cost | DECIMAL(10,2) | YES | — | — | 해당 일정 단위의 예상 비용 (식비, 입장료, 교통비 등) |
| memo | TEXT | YES | — | — | 사용자가 직접 작성하거나 챗봇이 요약한 상세 메모 |
| status | VARCHAR(20) | NO | — | `proposed` | `proposed` (제안됨), `confirmed` (확정됨) |

## 3. 대화 세션 및 플래닝 도메인 (Sessions, Chat & Proposals)
하나의 여행(Trip) 아래에 다수의 세션이 열릴 수 있으며, 챗봇 대화 중 발생한 '제안'을 처리하는 구조를 포함합니다.

### `sessions` — 대화 세션
세션은 여행 확정 전 탐색 단계에서도 생성될 수 있으므로 `trip_id`는 NULL 허용입니다. 세션 소유자는 `session_participants`의 `role='master'`로, 팀/개인 구분은 연결된 `trips→teams` 구조로 판별합니다.

| 칼럼명 | 타입 | NULL | 키 | 기본값 | 설명 |
|--------|------|------|----|--------|------|
| session_id | VARCHAR(50) | NO | PK | — | 세션 고유 식별자 |
| trip_id | VARCHAR(50) | YES | FK | — | 연결된 여행 (`trips` 참조) — 여행 미확정 시 NULL |
| title | VARCHAR(255) | YES | — | — | 세션 이름 (예: "오사카 2일차 맛집 논의") |
| is_manual_title | BOOLEAN | NO | — | false | 사용자가 직접 타이틀을 수정한 경우 true. LLM이 자동 갱신하지 않음 |
| topic | TEXT | YES | — | — | LLM이 추출한 현재 세션의 핵심 대화 주제 |
| context_summary | TEXT | YES | — | — | 이전 대화들에 대한 LLM의 누적 문맥 요약 |
| is_active | BOOLEAN | NO | — | true | 세션 활성화 여부 |
| created_at | TIMESTAMPTZ | NO | — | NOW() | 생성 시각 |
| updated_at | TIMESTAMPTZ | NO | — | NOW() | 마지막 메시지 발생 등 갱신 시각 |

### `session_participants` — 세션 내 참여자 및 상태
| 칼럼명 | 타입 | NULL | 키 | 기본값 | 설명 |
|--------|------|------|----|--------|------|
| session_id | VARCHAR(50) | NO | PK, FK | — | `sessions` 참조 |
| user_id | VARCHAR(40) | NO | PK, FK | — | `users` 참조 |
| role | VARCHAR(20) | NO | — | `viewer` | `master` (주도자), `participant` (참여자), `viewer` (관전자) |
| last_read_at | TIMESTAMPTZ | YES | — | — | 마지막으로 메시지를 읽은 시각 |

### `conversations` — 대화 내역 (텍스트, 파일, 제안 포함)
| 칼럼명 | 타입 | NULL | 키 | 기본값 | 설명 |
|--------|------|------|----|--------|------|
| message_id | VARCHAR(50) | NO | PK | — | 메시지 고유 식별자 |
| session_id | VARCHAR(50) | NO | FK | — | `sessions` 참조 |
| sender_id | VARCHAR(40) | YES | FK | — | 발신자 (NULL이면 시스템 또는 AI) |
| sender_type | VARCHAR(20) | NO | — | — | `user`, `ai`, `system` — DEFAULT 없음. 삽입 시 반드시 명시 (누락 시 AI 메시지가 user로 오염) |
| message_type | VARCHAR(20) | NO | — | `text` | `text`, `file`, `proposal` (일정 제안) |
| content | TEXT | NO | — | — | 텍스트 본문 (파일 참조 태그 포함 가능) |
| created_at | TIMESTAMPTZ | NO | — | NOW() | 발송 시각 |

### `proposals` — 제안-선택 방식 플래닝 제어 (AI가 특정 장소를 일정에 제안할 때)
| 칼럼명 | 타입 | NULL | 키 | 기본값 | 설명 |
|--------|------|------|----|--------|------|
| proposal_id | VARCHAR(50) | NO | PK | — | 제안 고유 식별자 |
| message_id | VARCHAR(50) | NO | FK | — | 이 제안이 포함된 원본 메시지 (`conversations` 참조) |
| target_day_id | VARCHAR(50) | NO | FK | — | 제안 대상 일자 (`trip_days` 참조) |
| place_id | VARCHAR(50) | NO | FK | — | 제안된 장소 (`places` 참조) |
| suggested_order | INT | YES | — | — | AI가 제안한 당일 일정 순서 (visit_order 후보값) |
| suggested_time | TIME | YES | — | — | AI가 제안한 방문 시간 |
| status | VARCHAR(20) | NO | — | `pending` | `pending` (대기중), `accepted` (수락됨), `rejected` (거절됨) |
| resolved_by | VARCHAR(40) | YES | FK | — | 수락/거절을 결정한 사용자 (`users` 참조) |
| resolved_at | TIMESTAMPTZ | YES | — | — | 수락/거절 결정 시각 |

### `session_markers` — 세션-장소 마커 연결 (지도 패널)
지도 패널의 마커는 좌표·이름 등 위치 데이터를 `places`에 저장하고, 이 테이블은 "어느 세션에서 어느 장소를 마커로 찍었는가"만 관리합니다. 마커가 일정으로 확정되면 해당 `place_id`를 그대로 `itinerary_items`에 사용할 수 있습니다.

| 칼럼명 | 타입 | NULL | 키 | 기본값 | 설명 |
|--------|------|------|----|--------|------|
| session_id | VARCHAR(50) | NO | PK, FK | — | `sessions` 참조 |
| place_id | VARCHAR(50) | NO | PK, FK | — | `places` 참조 — 좌표·이름 등 위치 데이터는 여기에 |
| marker_id | VARCHAR(50) | NO | UNIQUE(session_id, marker_id) | — | 프론트엔드 지도 라이브러리가 사용하는 마커 식별자 |

## 5. 알림 도메인 (Notifications)
Proposal 수락/거절, 세션 초대, 일정 변경 등 팀원 간 이벤트 알림의 영구 히스토리를 저장합니다. Redis Pub/Sub이 실시간 전달을 담당하고, 이 테이블이 읽지 않은 알림 뱃지 및 히스토리를 담당합니다.

### `notifications` — 사용자별 알림 히스토리
| 칼럼명 | 타입 | NULL | 키 | 기본값 | 설명 |
|--------|------|------|----|--------|------|
| notification_id | VARCHAR(50) | NO | PK | — | 알림 고유 식별자 |
| user_id | VARCHAR(40) | NO | FK | — | 알림 수신자 (`users` 참조) |
| type | VARCHAR(40) | NO | — | — | 알림 종류: `proposal_accepted`, `proposal_rejected`, `session_invited`, `trip_updated`, `member_joined` |
| reference_type | VARCHAR(20) | NO | — | — | 연관 객체 종류: `proposal`, `session`, `trip`, `team` |
| reference_id | VARCHAR(50) | NO | — | — | 연관 객체의 PK (proposal_id, session_id 등) |
| message | TEXT | NO | — | — | 사용자에게 표시될 알림 메시지 본문 |
| is_read | BOOLEAN | NO | — | false | 사용자가 읽었는지 여부 |
| created_at | TIMESTAMPTZ | NO | — | NOW() | 알림 생성 시각 |

## 6. Redis 키 형상 (빠른 세션 제어 및 큐 관리)

관계형 데이터베이스의 부하를 줄이고, 실시간 채팅 및 상태 동기화를 위해 Redis 구조를 아래와 같이 운영합니다.

| 키 패턴 | 데이터 타입 | TTL | 설명 및 용도 |
|---------|------------|-----|-------------|
**[인증]**

| 키 패턴 | 데이터 타입 | TTL | 설명 및 용도 |
|---------|------------|-----|-------------|
| `auth:refresh:{jti}` | String | MEM=7일, GST=24h | JWT Refresh Token의 `jti`(고유ID)를 키로, `user_id`를 값으로 저장. 로그아웃 시 삭제하여 서버 측 토큰 무효화를 구현. user_id가 아닌 jti를 키로 사용하는 이유: 다중 기기 로그인 지원 |
| `auth:blacklist:{jti}` | String | Access Token 남은 만료 시간 | 로그아웃 또는 탈취 의심 시 Access Token의 jti를 블랙리스트 등록. 매 요청 시 백엔드가 가장 먼저 조회하여 차단 |

**[세션/사용자 상태]**

| 키 패턴 | 데이터 타입 | TTL | 설명 및 용도 |
|---------|------------|-----|-------------|
| `user:GST:{uuid}` | Hash | 24h | 게스트 사용자 정보. PostgreSQL에 저장하지 않으므로 Redis가 유일한 저장소. TTL 만료 시 자동 소멸 |
| `user:{user_id}:sessions` | Set | MEM=48h, GST=24h | 해당 사용자가 소유한 session_id 목록. 세션 목록 API가 DB 조회 없이 빠르게 반환 |
| `session:{session_id}:meta` | Hash | MEM=48h, GST=24h | 세션 메타데이터: `owner`, `title`, `topic`, `context_summary`, `is_manual_title`. 활성 세션의 작업 저장소. 창 닫힘(flush) 시 PostgreSQL로 영구 이관. `mode` 필드 없음 — 팀/개인 구분은 trip→teams 구조로 판별 |
| `session:{session_id}:state` | String | 세션 수명 | 세션 현재 상태: `idle` / `processing`. LLM 추론 중 중복 요청 방지 및 UI 로딩 표시에 사용 |
| `user:presence:{user_id}` | String | 3분 | 웹소켓 하트비트(Ping-Pong)로 갱신. 온라인 여부 판단. 불필요한 푸시 알림 방지 |
| `team:{team_id}:active_users` | Set | 24h | 특정 팀 프로젝트에 접속 중인 실시간 사용자 목록 |

**[실시간 통신]**

| 키 패턴 | 데이터 타입 | TTL | 설명 및 용도 |
|---------|------------|-----|-------------|
| `session:{session_id}:pubsub` | Pub/Sub | 없음 | 채팅 메시지, Proposal 수락/거절 이벤트를 실시간으로 브로드캐스팅하는 채널 |
| `session:{session_id}:context` | List | 12h | LLM 참조용 문맥 캐싱. List 타입으로 RPUSH를 사용해 다중 코어 환경에서 원자적으로 밀어 넣음 (String 덮어쓰기의 Race Condition 방지) |

**[분산 락]**

| 키 패턴 | 데이터 타입 | TTL | 설명 및 용도 |
|---------|------------|-----|-------------|
| `lock:session:{session_id}:llm` | String | 1분 | SETNX로 획득. 다중 클릭/로드밸런싱 분산 환경에서 동일 세션의 LLM 추론 중복 실행 방지 |
| `lock:trip:{trip_id}:edit` | String | 5분 | 다중 접속 시 세부 일정(Itinerary) 동시 수정 충돌 방지 |

**[방어/속도 제한]**

| 키 패턴 | 데이터 타입 | TTL | 설명 및 용도 |
|---------|------------|-----|-------------|
| `rate:chat:{user_id}` | String (Int) | 1분 | 분당 챗봇 호출 횟수 카운팅(INCR). 무한 요청 루프 및 외부 API 과금 폭탄 방어 |

**[비동기 큐 및 캐시]**

| 키 패턴 | 데이터 타입 | TTL | 설명 및 용도 |
|---------|------------|-----|-------------|
| `queue:llm_tasks` | List | 없음 | 대화 요약, 장소 크롤링 등 비동기 워커 처리용 태스크 큐 |
| `cache:route:{place_a}:{place_b}` | String | 7일 | 두 장소 간 길찾기 API(폴리라인, 소요시간) 응답 JSON. 외부 API 중복 호출 방지 |
| `cache:weather:{region}:{date}` | String | 6시간 | 날씨 API 응답 결과 JSON |
| `cache:search:{query_hash}` | String | 1시간 | 크롤링 결과 JSON. query_hash는 쿼리 문자열의 MD5. 동일 검색어 반복 시 외부 크롤링 생략 |

**[알림]**

| 키 패턴 | 데이터 타입 | TTL | 설명 및 용도 |
|---------|------------|-----|-------------|
| `notify:unread:{user_id}` | String (Int) | 없음 | 읽지 않은 알림 건수. INCR/DECR로 관리. 뱃지 표시 시 DB 조회 없이 즉시 반환 |
