TravelArchive 기능 규칙서 (2026-05-11 기준)
코드베이스 직접 분석 기반 — 실제 구현된 동작만 기술

[ 1. 인증 ]

1.  회원 유형은 자체 계정(MEM)과 카카오 계정(KKO) 두 가지만 존재한다. 게스트 계정은 없다.

2.  자체 계정은 이메일 + 비밀번호로 가입한다. 이메일은 중복 불가이다.

3.  로그인 비밀번호를 5회 연속 틀리면 계정이 30분간 잠긴다.

4.  로그인 성공 시 Access Token(단기)과 Refresh Token(7일)이 발급된다.

5.  Refresh Token으로 새 Access Token을 재발급받을 수 있다.

6.  로그아웃 시 Refresh Token이 즉시 폐기되며, 이후 재발급이 차단된다.

7.  카카오 로그인은 OAuth 인가 코드 흐름을 사용한다. 카카오 계정은 이메일이 없을 수 있다.

8.  자체 계정(MEM)에 카카오 계정을 연동할 수 있다. 이미 다른 계정에 연동된 카카오 계정은 재연동 불가이다.

9.  카카오 계정으로 초대를 수행하려면, 상대방이 이메일이 있는 계정이어야 한다. 이메일 없는 카카오 계정은 검색 대상에서 제외된다.

10. 계정 삭제는 소프트 삭제(status='deleted')로 처리된다. 데이터는 DB에 유지된다.

11. 회원가입 및 카카오 최초 로그인 시 개인 팀이 자동으로 생성된다.


[ 2. 여행 계획(Trip) ]

12. 모든 여행 계획은 팀에 귀속된다. 사용자는 자신의 개인 팀을 통해 여행 계획을 소유한다.

13. 여행 계획에는 제목, 색상, 목적지, 시작일, 종료일, 상태(planning/completed 등)를 지정할 수 있다.

14. 각 사용자에게는 '기타(is_misc=true)' 여행 계획이 자동으로 존재한다. 이 계획은 삭제되지 않는다.

15. 여행 계획에 소속되지 않은 세션은 자동으로 기타 여행 계획에 귀속된다.

16. 여행 계획을 삭제하면, 해당 계획에 속한 모든 세션은 자동으로 기타 여행 계획으로 이동된다. 세션은 삭제되지 않는다.

17. 세션 설정에서 마스터는 해당 세션이 소속된 여행 계획을 변경할 수 있다.

18. 하나의 세션은 정확히 하나의 여행 계획에만 소속된다.

19. 홈 화면의 드롭다운에서 여행 계획을 선택하면 해당 계획 소속 세션만 필터링되어 표시된다.


[ 3. 세션 생명주기 ]

20. 세션을 생성한 사용자가 자동으로 마스터로 등록된다.

21. 모든 세션에는 마스터, 팀원, 봇이 기본으로 참여한다. 봇은 인원수 카운트에서 제외된다.

22. 세션 목록의 정렬 기준은 다음과 같다. 첫째로 팀원이 2명 이상인 세션(팀 세션)을 우선 표시한다. 둘째로 동순위 세션 간에는 최근 업데이트 순으로 정렬한다.

23. 팀원(비마스터)은 언제든지 자유롭게 세션에서 탈퇴할 수 있다.

24. 마스터는 팀원 전원을 퇴장시키고 개인 세션으로 전환할 수 있다.

25. 마스터가 세션에서 나가면 참여한 순서가 가장 오래된 팀원에게 마스터 권한이 자동 승계된다.

26. 마스터가 나갔을 때 남은 참여자가 없으면 세션이 비활성화(삭제)된다.

27. 세션 상태 정보(제목, 주제, 색상)가 변경되면 SSE를 통해 모든 구독자에게 실시간으로 전파된다.

28. 세션 제목은 LLM이 대화 내용을 바탕으로 자동으로 갱신한다.

29. 사용자가 세션 제목을 직접 변경하면 is_manual_title 플래그가 설정되며, 이후 LLM에 의한 자동 갱신이 중단된다.

30. 세션 색상의 기본값은 소속 여행 계획의 색상을 따른다.


[ 4. 임시 채팅 ]

31. 임시 채팅은 로그인 여부와 무관하게 사용 가능하다. 인증 없이도 접근할 수 있다.

32. 임시 채팅의 대화 내용은 DB나 Redis에 저장되지 않는다. 새로고침 시 대화가 사라진다.

33. 임시 채팅은 세션 목록에 나타나지 않으며, 여행 계획과도 연동되지 않는다.

34. 임시 채팅은 항상 LLM과 직접 대화한다. @BOT 접두사 없이도 LLM이 응답한다.


[ 5. 메시지 및 채팅 ]

35. 개인 세션(마스터 혼자)에서는 모든 메시지가 자동으로 @BOT으로 처리되어 LLM이 응답한다.

36. 팀 세션에서는 메시지 앞에 @BOT을 붙인 경우에만 LLM이 응답을 생성한다. 명시하지 않으면 LLM은 응답하지 않는다.

37. 팀 세션의 일반 메시지(비@BOT)는 SSE를 통해 같은 세션의 모든 구독자에게 실시간으로 전파된다.

38. @멘션(@닉네임)을 사용하면 해당 사용자에게 알림이 발송된다.

39. 대화 기록은 최신 40개를 기본으로 조회하며, offset 파라미터로 페이지네이션이 가능하다.

40. 파일을 업로드하면 세션과 사용자 ID별 폴더에 저장되며, 팀 세션에서는 SSE로 파일 첨부 사실이 실시간 전파된다.

41. 대화 기록은 텍스트 파일(.txt)로 다운로드할 수 있다.

42. 세션에 접속 중인 사용자 목록은 SSE 큐를 기반으로 추적한다. 세션 창을 열지 않은 참여자에게는 new_message 알림이 자동으로 전송된다.

43. 메시지를 읽으면 last_read_at이 갱신되어 미읽음 카운트에서 제외된다.

44. 미읽음 메시지 수는 세션 목록 조회 시 함께 반환된다.


[ 6. LLM 세션 컨테이너 ]

45. 각 활성 세션은 서버 메모리의 SessionContainer와 1:1로 연결된다.

46. SessionContainer는 역할이 분리된 세 개의 LLM 노드를 가진다. 각각 대화 생성(Generation), 주제 추론(Topic), 맥락 요약(Summary)을 담당한다.

47. 메시지가 누적되어 버퍼 한계(기본 6개)에 도달하면 과거 메시지를 DB에 저장하고 LLM이 요약문을 생성하여 context로 보관한다.

48. LLM 응답 생성 시 개인화 정보(사용자 선호), 현재 주제, 누적 맥락 요약, 과거 대화, 현재 메시지를 모두 컨텍스트로 주입한다.

49. 세션 종료 또는 로그아웃 시 버퍼에 남은 메시지가 DB에 플러시된다.


[ 7. 초대 및 팀원 ]

50. 초대는 정확한 이메일 주소를 입력해야 한다. 이메일이 일치하는 사용자만 검색 결과에 표시된다. 자신은 검색 결과에서 제외된다.

51. 자기 자신을 세션에 초대할 수 없다. 이미 참여 중인 사용자는 재초대가 차단된다.

52. 초대 알림을 수락하면 즉시 세션 참여자로 등록된다.

53. 초대 알림을 거절(dismiss)하면 해당 알림이 읽음 처리된다.


[ 8. 알림 ]

54. 알림 유형은 세션 초대(session_invite), @멘션(mention), 새 메시지(new_message)를 포함한다.

55. 알림은 SSE 스트림을 통해 실시간으로 전달된다.

56. 알림은 DB에 최대 30일 보관된다.

57. 알림을 읽음 처리하거나 전체 삭제하면 DB에서 즉시 제거된다.

58. 현재 세션을 열어 구독 중인 사용자에게는 new_message 알림이 발송되지 않는다.


[ 9. 지도(Map) ]

59. 지도 마커는 세션별로 Redis에 저장된다(TTL 24시간). 플러시 시 Postgres로 영구 저장된다.

60. 마커는 marker_id, 위도(lat), 경도(lng), 제목(title)으로 구성된다.

61. 마커를 추가하거나 삭제할 수 있다. 같은 marker_id의 마커는 덮어쓴다.

62. 폴리라인 경로는 마커 ID의 순서 목록으로 저장된다. 실제 좌표는 마커에서 참조한다.


[ 10. 일정 및 메모 ]

63. 일정(Plan)과 메모(Memo)는 날짜(date_key)별로 세션에 귀속되어 Redis에 저장된다.

64. 캘린더 인디케이터는 연도와 월을 기준으로 조회하며, 일정이 있는 날짜를 표시한다.

65. 여행 기간(trip_range)은 범위 목록 형태로 Redis에 저장된다.


[ 11. 사용자 설정 ]

66. UI 설정(테마, 폰트, 투명도, 사이드바 너비, 알림 설정)은 user_preferences.ui_settings에 JSONB로 저장된다.

67. 설정은 부분 업데이트를 지원하며, notifications와 같은 서브키는 기존 값과 딥머지된다.

68. 서비스 로드 시 저장된 설정이 자동으로 적용된다. 미로그인 상태에서는 기본값이 사용된다.

69. AI 스타일 설정(말투, 이모지 사용, 헤더 사용, 커스텀 지시사항)은 user_preferences.style에 저장된다.

70. 여행 취향 설정(여행 스타일, 페이스, 숙박 선호, 음식 선호, 알레르기, 최대 이동거리, 반려동물, 장애 정보)은 user_preferences.travel에 저장된다.

71. 프로필(닉네임, 소개, 추가 연락처)은 user_profile에 저장된다.


[ 12. 실시간 이벤트(SSE) ]

72. 세션 이벤트 스트림은 각 클라이언트 연결마다 독립된 Queue를 할당하여 관리한다. Queue 크기 한계(100)를 초과하면 메시지는 무시된다.

73. 25초마다 keepalive 이벤트를 전송하여 연결을 유지한다.

74. 클라이언트 연결 종료 시 해당 Queue가 구독 목록에서 자동으로 제거된다.

75. LLM 봇 응답은 트리거한 사용자에게는 스트리밍(문자 단위)으로 전달되고, 다른 참여자에게는 완성된 텍스트를 SSE로 브로드캐스트한다.

76. 타이핑 인디케이터 API는 엔드포인트가 존재하나 현재 동작하지 않는다(비활성 상태).


[ 13. 관리자 ]

77. 관리자 권한은 test@test.com 이메일 계정 또는 user_id가 ':admin'으로 끝나는 계정에 부여된다.

78. 관리자는 전체 사용자 목록(최대 200건)과 현재 메모리상 활성 세션 목록을 조회할 수 있다.
