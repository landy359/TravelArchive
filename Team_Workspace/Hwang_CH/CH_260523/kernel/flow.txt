# [Architecture & Flow Update] SDB - PPL 검색 파이프라인 및 데이터 흐름 개선

## 1. 개요 및 목적

이번 업데이트의 핵심은 "sdb 노드에서 파악한 사용자의 세부 의도(키워드)가 최종 LLM까지 유실 없이 전달되도록 파이프라인을 연결하는 것"입니다.

기존에는 DB에 존재하지 않는 사용자 요청(예: "서귀포 카약")이 발생할 경우 해당 의도가 소실되는 문제가 있었습니다. 이를 해결하기 위해 `QUST.PPL` 필드를 릴레이 채널로 활용하여, sdb가 찾지 못한 키워드를 ppl(Perplexity) 노드로 넘겨 외부 검색으로 보완하는 구조를 완성했습니다.

---

## 2. 데이터 파이프라인 흐름 (Data Flow)

`QUST` 객체를 중심으로 각 노드가 데이터를 순차적으로 채우고 소비합니다.

* **sdb 노드:** 사용자의 자연어 분석 후 DB 검색 진행. 매칭되지 않은 키워드는 자연어 검색 요청 문장으로 변환하여 `qust.PPL`에 저장.


* 
**ddb 노드:** 날씨 등 동적 데이터 수집.


* 
**ppl 노드:** `qust.PPL`에 텍스트가 있다면 이를 읽어 Perplexity에 검색을 요청하고, 그 결과(응답 텍스트)를 다시 `qust.PPL`에 덮어쓰기.


* 
**최종 LLM:** 완성된 `QUST.to_dict()` JSON 데이터를 전달받아 최종 일정을 계획.



---

## 3. 주요 모듈별 역할 및 변경 사항

### 🔹 `protocol.py` (통신 규약)

* 
`QUST` 데이터 클래스에 `days`(int) 필드를 추가했습니다.


* 
`days`는 sdb가 추출하며, 값이 없을 경우 기본값 0(이후 1로 처리됨)을 갖습니다.


* LLM으로의 JSON 직렬화를 위해 `to_dict()` 메서드에 `"days": self.days`를 명시적으로 추가했습니다.


* 초기 기획되었던 `unmet_keywords` 필드는 롤백되었으며, 대신 `PPL` 필드를 통합 채널로 활용합니다.



### 🔹 `sdb.py` (위치 기반 장소 검색 에이전트)

자연어 텍스트를 입력받아 좌표를 구하고, DB 검색 및 미매칭 키워드를 처리하는 메인 두뇌 역할을 합니다.

1. 
**의도 추출 (`_extract_intent`):** LLM을 1회 호출하여 `region`, `days`, `keywords`를 JSON 형태로 추출합니다.


2. 
**앵커 설정 (`_resolve_anchor`):** 3-Tier 방식(사전 정의 좌표 -> DB 검색 -> 랜덤 선택)을 통해 중심 좌표(lon, lat)를 확정합니다.


3. 
**메인 쿼리 (`_MAIN_QUERY`):** 중심 좌표 반경 8km 내에서 키워드 필터링(name, sub_category, alias)을 수행합니다. 카테고리별 균등 추출을 위해 `ROW_NUMBER() OVER (PARTITION BY ...)` SQL 기법을 사용합니다.


4. 
**미매칭 쿼리 (`_UNMET_QUERY`):** DB 검색 후 매칭되지 않은 키워드만 뽑아내어 Perplexity 검색용 자연어 문장("~ 근처에서 ~ 알려주세요")으로 포맷팅합니다.



### 🔹 `db_connector.py` (PostgreSQL 인프라)

도메인 로직 없이 순수하게 DB 연결과 쿼리 실행을 책임집니다.

* 
**비동기 처리:** `asyncio.to_thread`와 Connection Pool을 결합하여 I/O 블로킹을 방지합니다.


* 
**공간 쿼리:** PostGIS의 `ST_DWithin`과 `ST_Distance`를 활용한 `select_nearby` 메서드를 제공합니다.


* 
**보안 방어:** `_safe_id`를 통한 테이블/컬럼명 정규식 화이트리스트 검증과 `sql.Placeholder`를 통해 SQL Injection을 원천 차단합니다.
