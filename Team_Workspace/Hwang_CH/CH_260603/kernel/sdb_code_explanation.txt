[제주 여행 플래너 sDB 노드(sdb.py) 코드 상세 설명서]

본 설명서는 개편된 제주도 정적 장소 DB 조회 노드(sdb.py)의 아키텍처, 핵심 로직 및 SQL 쿼리 최적화 포인트를 상세히 기술합니다.

본 패치의 핵심 철학은 "파이썬 시스템은 신선하고 다양한 재료(장소 풀)를 넉넉하게 수급하고, 복잡한 요리(일정 조립 및 최종 판단)는 전적으로 LLM에게 위임한다"는 분리 원칙에 기반합니다.

────────────────────────────────────────────────
1. 구조적 개요 (Architecture Overview)
────────────────────────────────────────────────
sDB 노드는 사용자의 자연어 입력(질의 및 세션 주제)을 분석하여, 이에 부합하는 제주도 내 정적 장소 데이터를 PostGIS 공간 데이터베이스에서 효율적으로 인덱싱하고 파이썬 단에서 다양성을 보정하여 차세대 LLM 라우터에 제공하는 가교 역할을 수행합니다.

주요 데이터 흐름:
  [사용자 메시지 / 주제] ➔ LLM 의도 추출 ➔ 앵커(중심 좌표) 설정 ➔ PostGIS 반경 8km 거리순 조회 ➔ 카테고리별 파이썬 랜덤 샘플링 ➔ 미매칭 키워드 추출 ➔ QUST 객체 반환

────────────────────────────────────────────────
2. 주요 상수 및 파라미터 (Constants & Parameters)
────────────────────────────────────────────────
• CATEGORIES: ("숙소", "관광지", "식당", "카페", "사진스팟")
  - 기존 3개 카테고리에서 '숙소'와 '사진스팟'을 공식적으로 추가하여 총 5개 메인 카테고리 풀을 관리합니다.
• RADIUS_M: 8000.0 (8km)
  - 설정된 중심점을 기준으로 현실적인 이동 동선을 고려한 반경 8km 공간 필터링 표준 거리입니다.
• LIMIT_NORMAL (30개) / LIMIT_PHOTO (5개)
  - 데이터베이스 풀에서 1차로 긁어올 카테고리별 최대 개수입니다. 리소스를 효율적으로 제어하기 위해 사진스팟은 5개, 타 카테고리는 30개로 차등 제한합니다.
• SAMPLE_TARGETS
  - 식당(10), 관광지(10), 카페(6), 사진스팟(5), 숙소(4) 개로 설정되어 LLM에게 최종 전달됩니다. 매번 동일한 거리순 장소만 추천되는 단점을 극복하고, '다시 생성' 시 다양성을 확보하는 핵심 난수 장치입니다.

────────────────────────────────────────────────
3. 핵심 SQL 쿼리 상세 분석 (SQL Queries Analysis)
────────────────────────────────────────────────
(1) _ANCHOR_QUERY
    - 추출된 지역 지명(region)을 바탕으로 places 테이블에서 일치하는 장소의 위경도(geom)를 1건 찾아 중심점으로 삼습니다.

(2) _MAIN_QUERY (PostGIS 고성능 다중 필터링 및 차등 Limit 쿼리)
    - EXISTS 서브쿼리 최적화: 사용자가 입력한 keywords가 존재할 경우, places 테이블의 'name', 'sub_category' 뿐만 아니라, 1:N 관계에 있는 'alias(별칭)' 테이블을 EXISTS 절로 스캔하여 일치하는 장소를 전부 매칭합니다. 이를 통해 '카페'가 '식당' 카테고리에 오분류되어 있더라도 별칭 및 서브 카테고리 조회를 통해 완벽하게 검색망에 포착됩니다.
    - ROW_NUMBER() Window 함수 적용: `ROW_NUMBER() OVER (PARTITION BY m.main_category ORDER BY ST_Distance(...) ASC)` 구조를 사용하여, 카테고리별로 정렬된 거리 순위를 실시간으로 매깁니다.
    - WHERE 절 차등 조건 적용: 사진스팟은 5등까지, 그 외 카테고리는 30등까지만 하위 결과셋(ranked_places)에서 필터링하여 가져옴으로써 네트워크 및 DB 부하를 원천 차단합니다.
    - string_agg() 활용: 1:N 관계인 별칭(alias) 데이터들을 하나의 문자열로 묶어(`치킨, 통닭, 프라이드`) 파이썬 객체에 깔끔하게 매핑합니다.

(3) _UNMET_QUERY (미매칭 키워드 역추적)
    - 사용자가 요구한 키워드 배열 중, 반경 8km 내 장소의 이름/서브카테고리/별칭 어디에도 걸리지 않는 '실패 키워드'를 역으로 추출합니다. 이 미매칭 리스트는 차후 Perplexity(PPL) 노드로 넘어가 외부 실시간 웹 검색을 트리거하는 문장으로 가공됩니다.

────────────────────────────────────────────────
4. 실행 파이프라인 흐름 (Method Run Logic)
────────────────────────────────────────────────
Step 1: _extract_intent
  - LLM을 1회 호출하여 대화 문맥 속에서 지명, 일수, 키워드 배열을 구조화된 JSON으로 안전하게 파싱합니다. 파싱 에러 발생 시 시스템 셧다운을 방지하는 multi-tier 예외 방어 처리가 적용되어 있습니다.
Step 2: _resolve_anchor
  - 추출된 지명이 하드코딩된 거점(`REGION_CENTERS`)에 있다면 즉시 좌표를 반환하고, 없다면 DB 검색을 수행하며, 이마저도 실패하면 제주시/서귀포시 중 무작위로 기준점을 세우는 Fallback 매커니즘을 갖춥니다.
Step 3: 데이터베이스 쿼리 및 파이썬 단 2차 샘플링
  - DB에서 퍼온 풍부한 장소 풀(최대 125개)을 `defaultdict`를 이용해 카테고리별로 분류한 뒤, `random.sample`을 통해 최종 타겟 비율에 맞춰 무작위 추출합니다.
Step 4: 데이터 패킹 및 반환
  - 샘플링된 데이터를 프로토콜 객체 포맷인 `sDB_Item` 리스트로 변환해 `qust.sDB`에 주입하고, 미매칭 자연어 질문지 작성을 완료하여 라우터의 다음 노드로 토스합니다.

────────────────────────────────────────────────
5. 본 패치 코드가 가져오는 시스템적 이점
────────────────────────────────────────────────
1. 데이터 분류 오류 극복: 카카오/네이버 API 등 외부 연동 및 원천 데이터의 카테고리 오분류 문제를 서브 카테고리 및 다중 별칭 조인을 통해 파이썬 단에서 깔끔하게 흡수 및 보정합니다.
2. 시스템 부하 감소: 가중치 스코어 연산을 데이터베이스나 복잡한 파이썬 루프 내에서 수행하지 않고, 순수 거리 기반 공간 인덱싱(PostGIS)과 파이썬 내장 random 모듈만 사용하여 아키텍처가 매우 가볍고 직관적입니다.
3. LLM 환각 방지 및 자유도 보장: 고정된 일정 뼈대를 제공하는 대신, 검증된 좌표와 주소를 가진 풍부한 '장소 재료'만 엄선하여 넘겨줌으로써 LLM의 창의적 일정 조립 능력을 극대화하고 존재하지 않는 장소를 지어내는 할루시네이션을 완벽하게 방지합니다.
