sdb.py 수정사항

다중 앵커(순환 링) 도입: 일정(Day)에 따라 
탐색 기준점(앵커)이 제주시 ➔ 동부 ➔ 서귀포시 ➔ 서부 순으로 제주도를 한 바퀴 돌며 이동

동부/서부 거점 랜덤화: 동부나 서부 권역 내에서도 세부 거점(예: 애월, 한림, 모슬포)을 
무작위로 선택하여 매번 다른 동네의 장소들이 추천되도록 다양성을 확보

일차별 데이터 구조화: 데이터를 단순히 한 바구니에 담지 않고, 
1일차 장소들, 2일차 장소들처럼 일자별 딕셔너리로 묶어서 반환


port3.py 수정사항

프롬프트 조립 방식 변경: sdb.py가 넘겨준 일자별 장소 데이터를 읽고, LLM에게 
[1일차 앵커 주변 장소 후보] ... [2일차 앵커 주변 장소 후보] ... 형태로 깔끔하게 텍스트를 포매팅하여 전달

할루시네이션(환각) 방어: LLM이 일정을 다 짠 후, 시스템이 가진 실제 DB 데이터(이름, 주소, 좌표)와 
맵핑하기 위해 다차원 딕셔너리 데이터를 평탄화(Flatten)하여 검증하는 로직이 추가

ppl.py 수정사항

미매칭 키워드 연동: sdb.py에서 반경 8km 내에 매칭되는 장소가 없어 
실패한 키워드들(예: 특정 오션뷰 카페)을 전달받아, Perplexity에게 추가 집중 검색을 요청

일차별 데이터 포매팅: 외부 검색 시에도 일차별로 정리된 
DB 장소 리스트를 참고할 수 있도록 텍스트 구조를 업데이트



데이터 파이프라인 흐름도 (Data Flow)

사용자가 "3일 동안 오션뷰 카페랑 흑돼지 맛집 위주로 짜줘"라고 요청했을 때의 시스템 흐름

Step 1. 의도 파악 및 앵커 설정 (sdb.py)

LLM을 통해 일수: 3, 키워드: [오션뷰, 흑돼지]를 추출

3일이므로 순환 링을 돌려 앵커 3개를 선택. (예: 1일차: 제주시, 2일차: 성산(동부 랜덤), 3일차: 서귀포시)

Step 2. 반경 검색 및 샘플링 (sdb.py)

각 앵커 반경 8km 내에서 장소를 검색

중복을 제거한 뒤, 포토스팟은 전부, 나머지 카테고리는 최대 6개씩 무작위로 선정

[출력 데이터 예시] ```json
{
"1": {"카페": [A카페, B카페], "식당": [C식당, D식당], "사진스팟": [E해변]},
"2": {"카페": [F카페, G카페], "숙소": [H호텔]},
"3": {"식당": [I식당]}
}

Step 3. 외부 검색 트리거 (ppl.py)

만약 1일차 제주시 반경에서 '오션뷰'를 만족하는 장소를 DB에서 못 찾았다면, 
qust.PPL에 "제주도 근처 오션뷰 관련 장소 검색해줘"라는 문장을 담아 Perplexity 검색을 실행

Step 4. LLM 프롬프트 주입 및 일정 생성 (port3.py)

수집된 1~3일차 장소 리스트와 날씨 정보를 하나의 거대한 프롬프트로 묶어 메인 LLM에 전달

LLM은 "아, 1일차에는 A카페랑 C식당이 앵커 근처에 있구나"라고 인지하고 주어진 재료 안에서만 동선을 꼬임 없이 조립
